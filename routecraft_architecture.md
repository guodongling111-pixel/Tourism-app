# RouteCraft（行程织）架构设计

## 1️⃣ 系统整体架构概览
```
┌──────────────────────────┐
│ 前端 Web │
│ React + TypeScript + │
│ TailwindCSS + Leaflet │
│ │
│ Mock 数据驱动（核心） │
│ ATTRACTIONS / FOOD_SPOTS │
│ cityCoords (src/App.jsx) │
└─────────────┬────────────┘
│
│ 本地状态计算 / 路线生成逻辑
▼
┌──────────────────────────┐
│ 行程生成逻辑层 │
│ 前端算法（最近邻优化） │
│ 行程拆分（按天/时段） │
└─────────────┬────────────┘
│
│ 小红书攻略输入
▼
┌──────────────────────────┐
│ 攻略解析模块（前端） │
│ 用户粘贴文本 + NLP/LLM │
│ 结构化提取候选行程卡片 │
└─────────────┬────────────┘
│
▼
┌──────────────────────────┐
│ 地图展示层 │
│ Leaflet 路线可视化 │
│ 时间轴展示 + 拖拽调整 │
└──────────────────────────┘
```


### 🧠 架构说明（当前状态）

- 当前为**纯前端驱动架构**
- 无独立后端 API 服务
- 无数据库（PostgreSQL / PostGIS 未接入）
- 无外部数据抓取系统
- 所有数据来自前端 Mock JSON（src/App.jsx）
- 路线计算与生成在前端完成
  

## 2️⃣ 模块设计

---

### 前端模块（React + TypeScript + TailwindCSS + Leaflet）
- 功能：
  - 用户输入目的地、住宿、游玩天数
  - 景点 / 餐厅选择
  - 小红书攻略文本导入与候选结果展示
  - 时间轴 + 地图路线展示
  - 行程拖拽调整
- 特点：
  - 组件化开发，易维护
  - TypeScript 提供类型检查，减少 bug
  - 当前使用 Leaflet 进行地图与路径展示（后续可接入高德 API）

---

### 后端模块（FastAPI + Celery + Redis）

#### 功能
- 提供 RESTful API 服务

- 行程生成服务：
  - 基于最近邻算法进行路线优化
  - 支持多日行程自动分配（按天拆分景点与餐饮）
  - 支持餐饮节点插入与时间段匹配（Morning / Afternoon / Evening）

- 小红书攻略解析服务：
  - 接收用户粘贴文本内容
  - 异步任务（Celery）进行 NLP / LLM 结构化解析
  - 提取景点、餐厅、Citywalk 路线与推荐权重
  - 输出候选行程卡片数据

- 餐饮与景点数据管理服务

---

#### 当前实现
基于用户粘贴的攻略文本进行内容解析与结构化处理：

- 对用户输入文本进行信息拆解（地点、行程、标签等）
- 构建标准化数据结构，用于后续行程规划与推荐

---

#### 架构设计（规划中）
为扩展数据来源能力，设计攻略数据扩展方案：

- 动态网页抓取（后续）：Selenium
- 静态内容解析（后续）：Requests + BeautifulSoup
- 异步任务调度：Asyncio 提升处理能力

> 当前阶段：仅支持用户输入文本解析，不接入外部爬虫能力

---

#### 特点
- Celery + Redis 异步架构，支持未来高并发任务处理
- 模块化设计，支持 AI 个性化推荐扩展
- 输出结构化行程数据（如 Markdown / JSON）

---

### 数据库模块（PostgreSQL + PostGIS）
- 功能：
  - 存储景点、餐厅、行程、地图坐标
  - 支持地理计算（距离 / 最近邻）

- 特点：
  - 商用成熟稳定
  - PostGIS 降低地理算法开发成本

---

### 部署与运维（Vercel + Serverless）
- 功能：
  - Vercel 一体化部署
  - Serverless API 服务
  - 自动 CI/CD
  - CDN 静态资源加速

- 特点：
  - 零运维上线
  - 自动扩缩容
  - 适合 MVP 快速迭代

---

## 3️⃣ 数据流说明

---

### 1. 用户操作流程

1. 用户输入基础信息 + 选择景点/餐厅 + 粘贴小红书攻略文本
2. 前端基于本地 Mock 数据（ATTRACTIONS / FOOD_SPOTS / cityCoords）展示与筛选
3. 调用行程生成 API（如启用后端）
4. 小红书攻略文本通过 NLP/LLM 解析为结构化数据
5. 返回行程结果
6. 前端展示时间轴 + 地图路线，并支持拖拽调整

---

### 2. 数据存储流程

- 当前数据全部为前端 Mock：
  - ATTRACTIONS（景点）
  - FOOD_SPOTS（餐厅）
  - cityCoords（坐标）

- 数据定义在 `src/App.jsx`

- 行程结果仅前端临时计算与缓存（无持久化）

---

### 3. 外部数据扩展流程（Future Feature）

- 后续可扩展：
  - 后端 API 数据服务
  - PostgreSQL + PostGIS 数据库存储
  - Celery 异步数据处理任务

- 当前版本：
  - 不支持爬虫
  - 不使用 Selenium / Requests
  - 不接入外部数据源

---

## 4️⃣ 技术选型原则

1. 优先主流技术栈，降低学习与维护成本  
2. 工具成熟稳定，保证商用可靠性  
3. MVP 优先，支持快速上线与迭代  

---

## 5️⃣ 可扩展点

- AI 个性化推荐（用户偏好建模）
- 行程分享（生成可视化卡片）
- 多城市扩展（数据与地图体系扩展）

---

## 6️⃣ 总结

- 前后端分离架构清晰
- 支持异步任务处理与未来 AI 扩展
- 数据结构标准化，便于扩展数据库化
- 当前为 MVP 版本，轻量但具备可扩展性
